# Design Brief: PJ006 Portfolio Dashboard

> PM -> Designer 인수인계 문서 | 작성일: 2026-02-23

---

## 1. PRD 핵심 요구사항 요약

### 해결할 문제
- 6개 프로젝트가 목업 데이터(example.com)로 채워져 있어 실제 포트폴리오로 기능하지 않음
- 사이트 오너가 누구인지 알 수 없음 (프로필 섹션 부재)
- 기술 스택 정보가 없어 기술적 역량을 파악할 수 없음
- 카드 클릭 시 외부로만 나가고, 사이트 내에서 프로젝트를 깊이 볼 방법이 없음

### 핵심 목표
방문자가 **30초 안에** "이 사람이 어떤 프로젝트를 만드는 개발자인지" 파악 가능

### P0 신규 기능 (디자인 필요)
- P0-5: 프로필 섹션 (이름, 소개, 링크)
- P0-8: 카드에 기술 스택 태그 표시

### P1 신규 기능 (디자인 필요)
- P1-1: 프로젝트 상세 페이지 (/projects/[id])
- P1-4: 반응형 검증 및 개선
- P1-5: 호버/인터랙션 개선

---

## 2. 디자인 제약 및 결정

### 반드시 지켜야 할 것
1. **기존 Neumorphism 디자인 시스템 유지** — 색상 토큰, 그림자 시스템, 모서리 반경 모두 기존 그대로
2. **과한 장식 지양** — 깔끔하고 읽기 쉬운 것이 최우선
3. **정적 사이트** — 백엔드 없음, JSON 데이터 기반, 심플하게 유지
4. **기존 5개 컴포넌트와의 일관성** — 새 컴포넌트도 동일한 시각 언어 사용

### 디자인 결정 사항
- 프로필 섹션: Header 아래에 Neumorphism 카드로 배치 (Header 내부 통합 아님)
- 기술 스택 태그: 카테고리 태그와 동일한 nm-pressed 스타일
- 상세 페이지 최대 너비: max-w-3xl (메인 페이지 max-w-5xl보다 좁게)
- 카드 링크: 외부 -> 내부 라우팅으로 변경 (외부 링크는 상세 페이지에서 제공)

## 3. 미해결 질문

1. **프로필 데이터 위치**: profile.json 별도 파일 vs projects.json 내 포함 vs 하드코딩
   - 제안: `src/data/profile.json` 별도 파일 (Tech Lead가 결정)
2. **썸네일 이미지 (P1-2)**: 실제 스크린샷 사용 여부 - 현재는 이니셜 그래디언트 유지. 스크린샷이 준비되면 교체 가능한 구조로 설계
3. **다크모드 (P2-1)**: 현재 스코프 밖이나, 색상 토큰을 CSS 변수로 관리하고 있어 향후 확장 용이
